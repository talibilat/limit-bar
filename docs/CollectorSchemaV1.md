# LimitBar Local Usage Collector Schema v1

`limitbar-collect` and `CollectorWriter` are the supported interfaces for local usage producers.
They validate and write normalized token usage to local JSONL files.
They do not discover usage, parse native provider logs, proxy provider requests, or upload events.

## Domain Model

A usage event is an immutable record of the token quantities attributable to one logical completed operation.
`inputTokens` and `outputTokens` are per-event deltas, not cumulative provider counters or snapshots.
The event timestamp is when the usage occurred, preferably the operation completion time, rather than when the collector happened to write it.

A provider is the service responsible for the usage, not a client application, account, project, or producer process.
A custom source is a user-configured LimitBar ingestion channel identified by its stable UUID.
Source identity is a validated same-user claim, not cryptographic authentication.

## Request Schema

Each request is one UTF-8 JSON object of at most 16 KiB.
The exact allowed fields are:

| Field | Requirement |
| --- | --- |
| `schemaVersion` | Required integer and exactly `1`. |
| `eventID` | Required UUID generated by the producer and reused only when retrying the identical event. |
| `provider` | Exactly one identity field is required. Allowed values are `anthropic`, `azureOpenAI`, and `openAI`. |
| `customSourceID` | Exactly one identity field is required. This is the configured custom source UUID. |
| `timestamp` | Required ISO 8601 usage time. A timestamp more than five minutes in the future is rejected. |
| `model` | Required non-empty label, at most 256 UTF-8 bytes, with no control characters. |
| `deployment` | Optional only for `azureOpenAI`, with the same limits as `model`. |
| `inputTokens` | Required non-negative per-event token quantity. |
| `outputTokens` | Required non-negative per-event token quantity. |

Unknown fields are rejected rather than ignored.
The schema provides no fields for prompts, source code, responses, terminal data, request bodies, raw provider payloads, credentials, or arbitrary metadata.
Producers must not place such content into `model` or `deployment` labels.

Provider example:

```json
{"schemaVersion":1,"eventID":"fa2d37c5-1c49-49c8-88c4-6eebe339c6c7","provider":"azureOpenAI","timestamp":"2026-07-12T10:00:00Z","model":"gpt-4o","deployment":"production-chat","inputTokens":100,"outputTokens":20}
```

Custom-source example:

```json
{"schemaVersion":1,"eventID":"65f4c01b-d803-4609-bfaa-6a20eb65c132","customSourceID":"9598575e-259b-47df-9f34-f161c9015e65","timestamp":"2026-07-12T10:00:00Z","model":"local-model","inputTokens":100,"outputTokens":20}
```

## Compatibility

Collector v1 output includes the flat fields consumed by LimitBar's existing built-in and custom JSONL importers.
Those importers remain permissive for compatibility, but permissive importer input is not necessarily a valid collector v1 request.
The collector is the strict supported producer boundary and has no pass-through JSON or metadata option.

Provider events default to `~/Library/Application Support/LimitBar/usage-events.jsonl`.
The caller must submit a custom-source event to the exact file configured for that source and supply the configured source UUID as `customSourceID`.
Neither the reusable writer nor the CLI can authenticate the relationship between a custom-source UUID and a caller-supplied path.

## Idempotency And Conflicts

The producer must generate and persist an opaque UUID before submission.
It should use a random UUID that does not encode prompts, code, paths, credentials, or provider payloads.

Retrying an event already present in the active file returns `duplicate` without appending it or consuming additional rate capacity.
Reusing an existing UUID with different event content fails with `eventIDConflict`.
Idempotency is bounded to events still present in the active file; it is not a lifetime global guarantee.

## Concurrency And Files

Cooperating collector writers serialize through an owner-only interprocess `flock` sidecar.
While holding that lock, the writer atomically replaces complete files, so readers observe either the previous coherent JSONL file or the next coherent JSONL file.
The guarantee does not cover processes that bypass the collector and write directly.

The active file, lock, and rate-state files use owner-only permissions.
Existing active files and sidecars must be regular files rather than symbolic links.
The same-user local threat model does not protect against a malicious process already running as the user.

## Limits

The defaults are:

- A request is at most 16 KiB.
- A label is at most 256 UTF-8 bytes.
- One output file accepts at most 120 new events in a rolling 60-second interval across cooperating writers.
- The active file is at most 10 MiB, below the importer's 100 MiB hard limit.
- Events more than five minutes in the future are rejected.
- Old events are allowed for legitimate backfills.

Duplicates and rejected requests do not consume additional rate capacity.
Accepted-event times are stored in a bounded owner-only sidecar.
If sidecar persistence fails, the writer attempts to restore the previous active file and reports failure.

## Rotation And Archives

When an append would cross the active-file cap, the collector archives the complete prior active file and atomically rewrites the active file with events from the preceding eight days plus the new event.
Eight days retains every possible current local Monday-to-Monday week across clock changes.
Malformed or unrecognized existing lines are retained rather than silently discarded.

Archives are not ingested by LimitBar.
By default, archives older than 30 days are removed before rotation, and the remaining archives plus the new archive may occupy at most 1 GiB.
The oldest complete archives are removed first when the byte cap requires it.
Archive pruning is independent retention maintenance, so already-expired or over-cap archives may be removed even if the subsequent append fails.

If retained active data itself exceeds the active-file cap, the append fails with `activeFileTooLargeAfterRetention` rather than dropping data.
Policy values may be changed through `CollectorWriter`, but the active-file cap must remain at or below the importer's 100 MiB limit.

## CLI

Run:

```sh
swift run --package-path LimitBarCore limitbar-collect --help
```

Always persist an event UUID before invoking the command and reuse it only for an identical retry.
