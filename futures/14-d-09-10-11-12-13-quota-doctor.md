# Quota Doctor

## Executive decision

**Do not release another menu-bar quota display.** That segment is already crowded by open-source utilities such as ClaudeBar, OpenTokenUsage and ai-quota.

Quota Doctor should instead be a provider-neutral forensic and forecasting engine. It should explain quota consumption, estimate when a rolling window will be exhausted, attribute consumption to projects or agents, detect unexpected changes, and export an evidence bundle suitable for a provider bug report.

This is a natural extension of your existing limit tracker. The small-project version can be a CLI and local dashboard backed by imported session logs. It becomes strategically useful when connected to Agent Tail and, later, Agent Flight Recorder.

## Problem statement

Developers increasingly use several coding-agent subscriptions and APIs at the same time. Each service exposes different limits:

- requests per minute;
- tokens per minute;
- requests per day;
- spend limits;
- rolling multi-hour usage windows;
- weekly or monthly subscription allowances;
- model-specific weights;
- account or organisation-wide caps;
- temporary capacity restrictions.

The user-facing number is often insufficient to answer operational questions:

- Why did the remaining budget drop so quickly?
- Which agent, project or model consumed it?
- Is the displayed percentage consistent with observed requests?
- When will the current window reset?
- Can the remaining budget finish a planned task?
- Did consumption behaviour change after a model or client update?
- Is an error caused by quota, provider capacity or client state?

GitHub reports illustrate this opacity. In March 2026, one Claude Code user reported a five-hour window exhausting in one to two hours with unchanged work. In June 2026, a Codex user supplied session-log evidence showing a large increase in the percentage consumed per token. Another Claude Code report described requests failing despite a displayed usage level of only 16%. These are user reports and can contain mistaken interpretations, but they demonstrate the need for evidence, not merely a gauge.

## What exists today

### Provider dashboards and headers

- OpenAI documents API [rate limits](https://developers.openai.com/api/docs/guides/rate-limits) and provides an [API Usage Dashboard](https://help.openai.com/en/articles/10478918-api-usage-dashboard).
- Anthropic documents [rate limits](https://platform.claude.com/docs/en/api/rate-limits), including request, token and spend controls.
- Google documents [Gemini API rate limits](https://ai.google.dev/gemini-api/docs/rate-limits).

These are authoritative for supported API products, but subscription-based coding assistants may expose different or less granular information.

### Open-source trackers

- [ClaudeBar](https://github.com/tddworks/ClaudeBar) monitors several coding-assistant quotas from a macOS menu bar.
- [OpenTokenUsage](https://github.com/PowerUserZ/OpenTokenUsage) tracks coding subscriptions in the Windows tray.
- [ai-quota](https://github.com/niederme/ai-quota) provides a macOS quota utility for Codex and Claude Code.

These prove demand for convenient visibility but also prove that a basic status display is not a strong gap.

### Cost and trace platforms

LLM observability tools already record token counts, latency, model calls and estimated cost. Their primary unit is usually a trace or API request, not a subscription usage window with uncertain provider weighting.

## Precise product gap

Quota Doctor should provide five capabilities that ordinary trackers do not reliably combine:

1. **Attribution:** map consumption to project, session, model, agent and operation.
2. **Forecasting:** estimate exhaustion and reset times with confidence intervals.
3. **Anomaly detection:** identify sudden changes in usage-per-token, usage-per-task or window behaviour.
4. **Evidence:** preserve source observations and generate a redacted incident report.
5. **Planning:** estimate whether the remaining budget can complete a selected workload based on historical runs.

The product must distinguish three kinds of information:

- **Measured:** returned by an official API, response header or local client event.
- **Calculated:** derived from measured observations.
- **Inferred:** estimated when the provider does not disclose the true capacity or weighting.

Never display an inferred number as if it were an official quota.

## Target users

- Developers running coding agents heavily throughout the day.
- Teams operating several model providers or accounts.
- Builders of multi-agent systems with bursty usage.
- Maintainers investigating rate-limit regressions.
- FinOps or platform engineers managing LLM cost and capacity.

## Jobs to be done

- “Tell me what consumed the last 20% of my window.”
- “Warn me before my agents start a task they cannot finish within the remaining budget.”
- “Show whether this week’s consumption differs materially from previous weeks.”
- “Separate provider capacity errors from account quota errors.”
- “Create a redacted evidence report containing timestamps, client versions and rate-limit events.”
- “Allocate a daily or session budget across 30 concurrent agents.”

## Proposed user experience

### Import and status

```bash
quota-doctor import ~/.codex/sessions --provider codex
quota-doctor import ~/.claude/projects --provider claude-code
quota-doctor status
```

Example:

```text
Provider     Window       Used   Reset          Burn rate      Confidence
Codex        primary 5h   72%    14:43 BST      9.8% / 10 min  measured
Codex        weekly       18%    Mon 00:00      0.7% / hour    measured
Claude Code  session 5h   41%    16:10 BST      3.1% / 10 min  inferred: medium
Gemini API   RPM          12/60  rolling minute 7 req / min    measured

Warning QD-A03
Codex primary usage per 1,000 observed input tokens is 8.4x above the
14-day median. Change began 2026-06-18 07:22. Client version unchanged.
```

### Explain a change

```bash
quota-doctor explain --provider codex --from "2026-06-18 07:00" --to "2026-06-18 08:00"
```

Output groups consumption by:

- project;
- session;
- model;
- agent;
- tool type;
- prompt/input, cached, reasoning and output tokens where available;
- provider window.

### Task planning

```bash
quota-doctor can-run ./plans/security-review.yaml
```

```text
Estimated requirement: 31%-48% of primary window
Remaining: 28%
Result: unlikely to finish before exhaustion
Options:
  1. reduce parallel reviewers from 12 to 6
  2. route summarisation agents to a lower-weight model
  3. defer non-critical repository indexing until the reset
```

Recommendations must be rules-based and explainable. The MVP should not automatically switch accounts or circumvent provider limits.

### Integrated diagnostics and alerts

Ticket 11 owns the privacy-safe diagnostic export schema and preview flow.
Quota Doctor may add its measured, calculated, and inferred findings only through that allow-listed export contract.
Ticket 12 owns local notification thresholds, lock-screen copy, and exact-window deduplication.
Quota Doctor consumes those alert rules rather than creating a second notification system.

## MVP scope

### Six-to-ten-hour prototype

- Import one provider’s local JSONL or structured session logs.
- Store observations in SQLite.
- Display window percentage, reset time and observed token counts.
- Calculate burn rate and projected exhaustion.
- Detect a simple usage-per-token ratio anomaly.
- Export a Markdown incident summary.

### Hardened MVP

- Provider adapters for at least two subscription clients and one API provider.
- A canonical window model.
- Attribution to project and agent through trace IDs.
- Confidence labels and source provenance.
- EWMA-based burn-rate forecast.
- Baseline comparison and change-point warnings.
- Local web dashboard or terminal charts.
- Integration with the local alerts from ticket 12.
- Integration with the privacy-safe diagnostic export from ticket 11.

### Explicit non-goals

- Circumventing limits, rotating accounts or evading provider controls.
- Scraping private web pages through brittle browser automation by default.
- Claiming to know undisclosed provider capacity.
- Replacing official billing records.
- Automatically purchasing credits or changing plans.
- Forecasting monetary cost without an explicit, versioned price source.
- Exposing raw prompts unless the user explicitly opts in.

## Canonical data model

### Observation

```json
{
  "observation_id": "01J...",
  "provider": "codex",
  "account_alias": "work-plus",
  "source": {
    "type": "local_session_event",
    "path": "redacted/session.jsonl",
    "source_confidence": "measured"
  },
  "timestamp": "2026-06-18T07:22:34Z",
  "trace_id": "run-44",
  "project_id": "payments-api",
  "agent_id": "reviewer-7",
  "model": "gpt-5.5",
  "usage": {
    "input_tokens": 23469,
    "cached_input_tokens": null,
    "reasoning_tokens": 48,
    "output_tokens": null
  },
  "windows": [
    {
      "window_id": "primary",
      "used_percent": 57,
      "resets_at": null,
      "provenance": "event.rate_limits.primary"
    }
  ]
}
```

### Window definition

```yaml
provider: codex
window_id: primary
kind: rolling
reported_capacity: null
reported_used_unit: percent
nominal_duration: 5h
reset_semantics: observed
weighting_model: undisclosed
```

### Derived metric

Every derived metric records its formula and input range:

```json
{
  "metric": "used_percent_per_1k_input_tokens",
  "value": 0.82,
  "formula_version": "ratio-v1",
  "inputs": ["obs-1", "obs-2"],
  "confidence": "medium",
  "limitations": ["cached tokens unavailable", "provider weighting undisclosed"]
}
```

## Forecasting approach

### Burn rate

For a simple window:

```text
instantaneous burn = change in used percentage / elapsed minutes
```

Smooth with an exponentially weighted moving average:

```text
EWMA_t = alpha * burn_t + (1 - alpha) * EWMA_(t-1)
```

Use a lower `alpha` for stable background workloads and a higher value for interactive sessions. Let users select conservative, balanced or responsive forecasting.

### Projected exhaustion

```text
minutes_remaining = remaining_percent / current_burn_per_minute
```

Only display this when the burn rate is positive and sufficiently observed. Include a range based on recent percentile rates rather than one precise timestamp.

### Usage-per-work ratio

Possible denominators:

- 1,000 input tokens;
- one model request;
- one completed task;
- one agent step;
- one accepted code change;
- one minute of active session time.

Token ratios alone can mislead when provider weighting includes model, caching, reasoning or service-tier factors. Compare several ratios.

### Change detection

MVP options:

- robust z-score using median and median absolute deviation;
- ratio to trailing median;
- cumulative sum or Bayesian change-point detection later.

An alert should show the exact comparison window and raw observations.

## Source adapters

Prioritise official and stable sources:

1. API response headers and documented usage endpoints.
2. Structured local client events.
3. OpenTelemetry spans emitted by the user’s runtime.
4. User-imported CSV or screenshots converted manually.
5. Browser automation only as an optional community adapter with a clear breakage and terms warning.

Each adapter should declare:

- supported versions;
- fields captured;
- whether authentication material is touched;
- confidence level;
- known missing fields;
- last verified date.

## Architecture

```text
Provider adapters / Agent Tail events
              |
              v
Observation normaliser
  timestamps | units | window IDs | provenance | redaction
              |
              v
SQLite event store
  immutable observations | derived metrics | baselines
              |
       +------+-------+
       |              |
       v              v
Analytics engine    Exporters
forecast/anomaly    CLI/TUI/HTML/JSON/report bundle
```

Use append-only observations. If an adapter later corrects an event, add a superseding record rather than rewriting history silently.

## Integration with your agent runtime

Add these attributes to each Agent Tail event:

```json
{
  "provider": "openai",
  "account_alias": "team-a",
  "model": "...",
  "quota_window_ids": ["primary", "weekly"],
  "usage_source": "provider_event",
  "budget_allocation_id": "security-review-run"
}
```

Quota Doctor can then allocate the observed delta across concurrent agents. When a provider only emits account-level percentage updates, attribution is approximate. Use proportional allocation based on token or request observations and label it inferred.

## Security and privacy

- Store account aliases, never credentials.
- Do not persist raw prompts by default.
- Hash project paths or replace them with configured names.
- Encrypt local storage optionally, especially on shared workstations.
- Redact session IDs if they could be support or authentication identifiers.
- Do not send reports automatically.
- Maintain a data-retention setting and one-command purge.
- Treat browser cookies and OAuth tokens as out of scope for the core.

## Testing strategy

### Fixtures

- stable consumption;
- bursty consumption;
- reset event;
- percentage counter moving backwards after reset;
- missing token data;
- account-level quota with concurrent agents;
- provider event arriving out of order;
- client-version change coinciding with ratio change;
- corrupted or partially written JSONL.

### Invariants

- raw observations are immutable;
- derived values can always identify their inputs;
- measured and inferred values have different labels;
- reset detection never converts a counter decrease into negative usage;
- one adapter cannot read files outside configured paths;
- export contains no configured secret patterns.

## Success metrics

- At least 90% of imported supported sessions produce a coherent window timeline.
- Forecast error under 20% for stable workloads with at least 30 minutes of observations.
- At least 70% of anomaly alerts are accepted as useful in pilot review.
- At least 30% of users connect project or agent attribution.
- Evidence reports reduce the time required to submit a provider issue or internal incident.

## Validation plan

Recruit 10 heavy coding-agent users. Ask them to install the prototype for one week and record:

- when they unexpectedly hit a limit;
- whether the tool predicted it;
- whether attribution matched their expectation;
- which raw source they trusted;
- whether they changed workflow based on the warning;
- whether the report contained enough evidence to contact support.

The strongest signal is not “the gauge looks good”. It is “I changed scheduling or identified a regression because of the evidence.”

## Main risks and mitigations

### Risk: provider formats change frequently

Mitigation: isolated adapters, version tests, adapter health status and community-maintained fixtures.

### Risk: undocumented weighting makes forecasts misleading

Mitigation: confidence labels, ranges, multiple denominators and explicit “provider weighting unknown” notices.

### Risk: the project competes directly with existing trackers

Mitigation: avoid menu-bar-first positioning. Lead with attribution, evidence and anomalies.

### Risk: local logs contain sensitive prompts

Mitigation: field allowlists, no prompt persistence by default and redaction before storage.

### Risk: users interpret a forecast as a guarantee

Mitigation: display an interval and observation count. Use language such as “likely” and “based on recent burn”.

### Risk: provider terms restrict extraction methods

Mitigation: prioritise official interfaces and user-owned local files. Keep browser adapters separate and document terms considerations.

## Open-source positioning

Recommended licence: Apache-2.0.

Repository promise:

> Explain where your AI coding quota went. Attribute usage, forecast exhaustion, detect sudden changes and export a redacted evidence report across providers.

Good contributor tasks:

- provider adapter;
- parser fixture;
- forecast model;
- notification integration;
- redaction rule;
- dashboard panel.

## Naming alternatives

- Quota Doctor
- Headroom
- Limit Lens
- BurnScope
- Quota Forensics

“Headroom” is attractive but likely crowded. Verify trademarks and package names.

## Go/no-go criteria

Proceed when:

- users can identify at least one previously opaque consumption event;
- two providers have stable adapters;
- project or agent attribution changes user behaviour;
- anomaly alerts remain useful after four weeks;
- users prefer the evidence report over screenshots of a quota bar.

Pause when:

- supported clients expose too little structured data for reliable analysis;
- users only want a status-bar percentage;
- maintaining private-format parsers consumes more effort than the product value.
